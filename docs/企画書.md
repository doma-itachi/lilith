LilithというRust製の爆速なニコニコ動画コメ付きダウンローダーを作りたい

CLIで`lilith https://www.nicovideo.jp/watch/sm45174902`のように実行すると
1. yt-dlpで動画のダウンロード
2. コメントのダウンロード
3. 動画の読み込み→コメント描画→出力
みたいな感じにしたいです

## コメント描画に関して
コメント描画にはnpmのNiconiComments(https://github.com/xpadev-net/niconicomments)というライブラリをRustに移植して使いたいと思っている

## 技術スタック案

### 方針
MVPでは「全部をRustネイティブで抱え込まない」方がいいです。
動画の取得と最終エンコードは実績のある外部ツールに任せて、Lilith本体は以下に集中します。

1. URL解析とジョブ管理
2. コメント取得と正規化
3. NiconiComments互換のコメント描画エンジン
4. ffmpegへのフレーム受け渡し

これならRustで作るべき価値のある部分に集中できます。

### 推奨スタック

#### 言語・ランタイム
- Rust stable
- Edition 2024
- 非同期実行は`tokio`

`yt-dlp`や`ffmpeg`のプロセス管理、ダウンロード待機、ログ収集を考えると`tokio`前提が扱いやすいです。

#### CLI
- `clap`: コマンドライン引数の定義
- `indicatif`: 進捗表示
- `inquire` or `dialoguer`: 必要なら対話UI

CLIツールとしての使い勝手はこの組み合わせで十分です。

#### エラー処理・ログ
- `anyhow`: アプリ全体のエラー集約
- `thiserror`: ドメイン固有エラー定義
- `tracing`
- `tracing-subscriber`

ライブラリ層は`thiserror`、CLI境界では`anyhow`でまとめる構成が素直です。

#### HTTP・データ処理
- `reqwest` + `rustls`: コメントAPIやメタデータ取得
- `serde`
- `serde_json`
- `quick-xml`: XML系レスポンスを扱う場合
- `url`: URL正規化

ニコニコ周辺はJSONとXMLの両方に触れる可能性があるので、この構成が無難です。

#### 外部ツール連携
- `yt-dlp`: 動画取得
- `ffmpeg`: エンコード、合成、音声付き出力
- Rust側では`tokio::process::Command`で呼び出し

ここは無理にRust実装へ置き換えない方がいいです。特にエンコード周りを自前で持つと開発コストが急に跳ねます。

#### コメント描画エンジン
- Lilith独自crateとして`lilith-render`を切る
- 座標計算・アニメーション・レイアウトはNiconiCommentsをRust移植
- 2D描画は`tiny-skia`
- 文字レイアウトは`cosmic-text`
- 色処理や座標計算は標準ライブラリ + 必要なら`glam`

`tiny-skia`はRustだけで完結しやすく、描画対象がコメントオーバーレイ中心なら十分実用的です。
日本語テキスト描画は`cosmic-text`を使った方が後で困りにくいです。

#### 動画合成
- Rustで各フレームのコメントオーバーレイをRGBAで生成
- `ffmpeg`へパイプして元動画と合成

この方式なら、Lilithはコメント描画に集中できて、最終的なエンコード品質は`ffmpeg`に任せられます。

### 推奨構成

#### バイナリ構成
- `lilith-cli`: CLI入口
- `lilith-core`: URL解析、ジョブ管理、設定
- `lilith-nico`: ニコニコ動画/コメント取得
- `lilith-render`: コメント描画
- `lilith-ffmpeg`: ffmpeg連携

最初からworkspaceで切っておくと、描画部分のテストと再利用がしやすいです。

#### 各ディレクトリの役割
- `crates/`: Rust本体。責務単位でcrateを分離する
- `assets/fonts/`: デフォルトフォントや描画検証用フォントを置く
- `assets/testdata/`: 小さめの検証素材。ユニットテスト寄り
- `fixtures/comments/`: コメント入力サンプル
- `fixtures/videos/`: E2Eや手元検証用の短尺動画
- `fixtures/snapshots/`: 描画結果の比較用画像
- `scripts/`: ローカル開発補助
- `tests/`: workspace横断の統合テスト
- `vendor/niconicomments/`: 移植元のTypeScript実装を参照用に保持

#### 現在のリポジトリに当てはめるなら
- `src/main.rs` は将来的に `crates/lilith-cli/src/main.rs` へ移す
- `ts/niconicomments/` は削除せず、まずは `vendor/niconicomments/` 相当の参照実装として扱う
- 企画書や設計メモは `docs/` に集約する

#### この構成にする理由
- CLI、ニコニコ連携、描画、ffmpeg連携の依存関係を分離しやすい
- `lilith-render` を単体でベンチ・スナップショットテストしやすい
- 移植元TypeScriptを本体コードと分けて保持できる
- 将来GUIやサーバーモードを追加しても `lilith-core` と `lilith-render` を再利用しやすい

#### 想定フロー
1. CLIでURLを受け取る
2. `yt-dlp`で動画を取得
3. ニコニコのコメントを取得して内部形式に正規化
4. `lilith-render`で時刻ごとのコメントオーバーレイを生成
5. `ffmpeg`で元動画に合成してmp4出力

### MVPでやること / 後回しにすること

#### MVPでやること
- 単一動画URL対応
- 通常コメント、上コメ、下コメ対応
- 基本的なフォント、サイズ、色、速度
- mp4出力
- 1080pまでの安定動作

#### 後回しでいいこと
- GPUレンダリング
- GUI
- プレイリスト/連続DL
- 高度なエフェクト完全再現
- 完全ネイティブの動画デコード/エンコード

### 重要な判断
NiconiCommentsのRust移植はこの企画の核ですが、動画のデコード/エンコードまでRustで抱えるのは分離した方がいいです。
Lilithの差別化は「爆速なコメント付き出力」と「CLI体験」にあるので、MVPでは以下の技術選定が最も筋が良いです。

- コア言語: Rust
- 動画取得: `yt-dlp`
- 動画合成: `ffmpeg`
- コメント描画: Rust移植した`NiconiComments` + `tiny-skia` + `cosmic-text`
- 非同期制御: `tokio`
