# Lilith 作業分割

## 方針
MVPでは以下の順で作る。

1. まず動く CLI パイプラインを作る
2. 次にコメント取得を安定させる
3. その上でコメント描画を Rust に移植する
4. 最後に速度最適化を入れる

GPU は最初から必須にしない。
先に `ffmpeg` のハードウェアデコード/エンコード対応を入れ、その後に必要なら描画 backend を GPU 化する。

## フェーズ 0: 足場固め

### 目的
- 開発を進められる workspace と開発手順を固定する

### 作業
- `Cargo workspace` の維持
- `README.md` を追加して開発手順を書く
- `docs/企画書.md` とこの `docs/作業.md` を運用ドキュメントにする
- `.gitignore` を整理する
- `vendor/niconicomments` を参照実装として固定する
- `vendor/niconicomments/.git` の扱いを決める

### 完了条件
- 初回 clone 後に `cargo check --workspace` が通る
- どの crate に何を書くかが明確

## フェーズ 1: CLI の骨格

### 目的
- `lilith <url>` で処理開始できるようにする

### 作業
- `lilith-cli` に `clap` を導入する
- 引数を定義する
- 想定オプションを決める
- URL バリデーションを入れる
- 出力先ディレクトリを指定できるようにする
- `--keep-temp`
- `--hwaccel auto|none|videotoolbox|nvenc|qsv|amf`
- `--font`
- `--quiet`

### 完了条件
- `lilith https://www.nicovideo.jp/watch/...` が解釈できる
- 不正入力時に適切なエラーを返す

## フェーズ 2: ジョブ・設定・一時ファイル管理

### 目的
- 1 回の実行に必要な状態を `lilith-core` で一元管理する

### 作業
- `AppConfig` を実装する
- `Job` を拡張する
- 作業ディレクトリの設計を決める
- 一時ファイル命名規則を決める
- 出力ファイル命名規則を決める
- ログレベルと進捗表示の責務を整理する

### 完了条件
- URL から 1 件のジョブを組み立てられる
- 中間ファイルの配置ルールが固定される

## フェーズ 3: `yt-dlp` 連携

### 目的
- 元動画を確実に取得する

### 作業
- `lilith-core` か `lilith-nico` に downloader 境界を定義する
- `yt-dlp` 実行ラッパを作る
- バイナリ存在確認を入れる
- 標準出力・標準エラーの収集を実装する
- ダウンロード済みファイルの検出を入れる
- 再試行ポリシーを決める

### 完了条件
- 指定 URL から動画ファイルを保存できる
- `yt-dlp` がない場合に明示的に失敗する

## フェーズ 4: ニコニコ動画メタデータ取得

### 目的
- 動画 ID に対して必要な API 情報を取得できるようにする

### 作業
- watch URL から動画 ID を安定抽出する
- 動画タイトル、長さ、コメント取得に必要な情報を集める
- `reqwest` + `serde` を導入する
- レスポンスの Rust モデルを定義する
- エラー種別を定義する

### 完了条件
- 対象動画のメタデータ取得ができる
- コメント取得に必要な情報が `Job` へ入る

## フェーズ 5: コメント取得と正規化

### 目的
- コメントソースを Rust 側の内部形式へ落とし込む

### 作業
- コメント API の入力形式を調査する
- JSON/XML のどちらを基準にするか決める
- `lilith-nico::comment` の内部モデルを定義する
- `posted_at`
- `vpos`
- `content`
- `mail`
- `premium`
- `owner`
- `layer`
- パーサを書く
- ソート順とフィルタリングルールを書く
- `vendor/niconicomments` の入力仕様との差分を洗う

### 完了条件
- 1 本の動画についてコメント一覧を内部形式へ正規化できる
- フィクスチャを使ったパーステストが通る

## フェーズ 6: NiconiComments 移植調査

### 目的
- TypeScript 実装を Rust の責務に分解する

### 作業
- `vendor/niconicomments/src` を責務単位で棚卸しする
- 以下を分類する
- 入力パース
- コメントモデル
- レイアウト
- 衝突回避
- スクロール計算
- 色・サイズ・位置ルール
- レンダラ依存処理
- Rust へ移植する優先順位を決める
- 移植対象外を決める

### 完了条件
- 「MVPで移植する機能」と「後回し機能」が分かれている
- `lilith-render` のモジュール責務が固まる

## フェーズ 7: CPU レンダラ MVP

### 目的
- まず CPU で正しく描画できる状態にする

### 作業
- `tiny-skia` を導入する
- `cosmic-text` を導入する
- 文字測定とテキストレイアウトを実装する
- 固定コメントの描画を実装する
- 流れるコメントの座標計算を実装する
- 上下固定コメントを実装する
- 縁取り、色、サイズ、透明度を実装する
- RGBA バッファ出力を作る

### 完了条件
- ある時刻のコメントオーバーレイ 1 フレームを生成できる
- サンプル入力に対して視覚的に破綻しない

## フェーズ 8: タイムラインと衝突回避

### 目的
- ニコニココメントらしい挙動を再現する

### 作業
- レーン管理を実装する
- 衝突回避ロジックを実装する
- コメントの生存時間を実装する
- 速度計算を実装する
- シーク時の再現性を確認する
- スナップショットテストを整備する

### 完了条件
- 同時大量コメントでも表示破綻が限定的
- 代表ケースで snapshot 比較が通る

## フェーズ 9: `ffmpeg` 合成

### 目的
- 元動画とコメントオーバーレイを結合して mp4 を出力する

### 作業
- `lilith-ffmpeg` にコマンド組み立てを実装する
- 入力動画情報を `ffprobe` または `ffmpeg` から取得する
- フレームレートと解像度の整合を取る
- RGBA フレームのパイプ入力を実装する
- 音声を維持したまま動画を再エンコードする
- 中間ファイルを掃除する

### 完了条件
- 元動画 + コメント入り動画の mp4 を出力できる
- 音ズレ、フレーム落ちが許容範囲

## フェーズ 10: ハードウェアエンコード対応

### 目的
- 低コストで最初の高速化を入れる

### 作業
- `--hwaccel` オプションを実装する
- macOS 向け `videotoolbox` 対応
- NVIDIA 向け `nvenc` 対応
- Intel 向け `qsv` 対応
- AMD 向け `amf` 対応
- 利用可能 codec の事前チェックを入れる
- 利用不可時に CPU へフォールバックする
- ベンチマークを取る

### 完了条件
- 環境ごとに利用可能な HW encode を自動または明示指定で選べる
- CPU only より明確に速いケースを確認できる

## フェーズ 11: GPU レンダラ設計

### 目的
- 必要になったときに描画自体を GPU 化できるようにする

### 作業
- `lilith-render` に backend 抽象を入れる
- `CpuRenderer`
- `GpuRenderer`
- backend 非依存の責務を分離する
- `layout`
- `timeline`
- `comment model`
- 依存境界を整理する
- `wgpu` 採用可否を決める
- GPU 転送コストの計測方法を決める

### 完了条件
- CPU backend と GPU backend を差し替え可能な設計になる
- GPU 化の前提調査が終わる

## フェーズ 12: GPU レンダラ実装

### 目的
- CPU がボトルネックになった場合に限り、描画処理を GPU へ逃がす

### 作業
- `wgpu` backend を実装する
- グリフ描画方法を決める
- テクスチャ atlas を設計する
- 縁取りの表現を決める
- GPU から `ffmpeg` への受け渡し経路を決める
- 読み戻しコストを測る
- CPU 版と品質比較をする

### 完了条件
- CPU backend より速いケースが実測で確認できる
- 品質劣化が許容範囲

## フェーズ 13: 品質保証

### 目的
- 壊れにくい CLI ツールにする

### 作業
- ユニットテストを書く
- 統合テストを書く
- E2E テストを書く
- snapshot テストを書く
- 大量コメントケースのベンチを作る
- 失敗時ログの改善を行う

### 完了条件
- 主要な正常系と異常系を自動確認できる
- 描画回帰を snapshot で検出できる

## フェーズ 14: 配布

### 目的
- 手元以外でも動かせる状態にする

### 作業
- 対応 OS を決める
- 必須依存のインストール手順を書く
- `yt-dlp` と `ffmpeg` の前提を README に明記する
- リリースビルド手順を書く
- 配布方法を決める
- Homebrew
- GitHub Releases

### 完了条件
- 他人がセットアップして CLI を実行できる
